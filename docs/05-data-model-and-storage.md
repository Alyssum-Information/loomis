Loomis stores **metadata** in SQLite and **bulk content** (audio, transcripts,
Markdown) on the filesystem. The DB is the source of truth for relationships and
job state; the filesystem holds the heavy bytes.

> Schemas below are illustrative (SQLite dialect). The implementation owns the
> authoritative DDL and migrations; treat this as the design intent.

## 1. On-disk layout

```
<data_dir>/                     # single configurable root (see 06)
  loomis.db                     # SQLite database (WAL mode)
  config.toml
  library/
    <device_slug>/<YYYY>/<MM>/<recording_id>.<ext>   # imported audio
  transcripts/<recording_id>.json   # full word-timestamped transcript + diarization
  diary/<YYYY-MM-DD>.md             # daily diary entry (+ sibling .json metadata)
  meetings/<meeting_id>.md          # meeting record (+ sibling .json metadata)
  voiceprints/                      # (optional) raw embedding blobs if not inlined
  staging/                          # in-flight copies, pre-verification
  quarantine/                       # copies that failed verification
  logs/
```

Rationale: large transcripts/embeddings keep the DB small and fast; Markdown on
disk is diffable, greppable, and trivially synced to cloud by rclone.

## 2. On-device registration file — `.loomis/device.json`

Written to the **root of the recorder's volume** at registration. Lets Loomis
recognize the device on any computer and carries the device's preferences.

```jsonc
{
  "schema": "loomis.device/v1",          // contract version, for forward-compat
  "device_id": "0f6c1e2a-...-uuid",       // stable id, generated by Loomis
  "name": "Sony ICD-TX660",
  "owner_speaker_hint": "kevin",          // whose voice this device mostly holds
  "registered_at": "2026-06-06T12:00:00+08:00",
  "loomis_version": "0.1.0",
  "audio_globs": ["**/*.wav", "**/*.mp3", "**/*.m4a"],
  "backup": {
    "auto_delete_after_backup": false,    // delete source only after verified copy
    "min_free_bytes_guard": 0
  },
  "transcode": {
    "policy": "keep_original",            // keep_original | transcode_keep | transcode_only
    "codec": "opus",
    "bitrate": "16k",
    "application": "voip"
  }
}
```

Validation: parsed (pydantic) on every connect. Unknown future `schema` versions
read defensively; missing optional keys fall back to global config. Read-only
volume → registration proceeds in the DB only; matching falls back to volume
serial/label. Rationale: [ADR-0009](adr/0009-device-registration-format.md).

## 3. Entity overview

```
devices ─< recordings ─< transcripts ─< segments ─ speaker
   │            │
   │            └─< jobs (pipeline steps)
speakers ─< voiceprints
diary_entries ─< diary_recordings >─ recordings
meetings ─< meeting_participants >─ speakers
diary_meeting_links (diary_entries ─ meetings)
```

## 4. Tables

### 4.1 `devices`
| Column | Type | Notes |
|--------|------|-------|
| id | TEXT PK | = `device_id` (UUID) |
| name | TEXT | human label |
| volume_serial | TEXT | fallback identity hint |
| owner_speaker_id | INTEGER FK→speakers | nullable |
| audio_globs | TEXT (JSON) | match patterns |
| auto_delete | INTEGER (bool) | source-cleanup policy |
| transcode_policy | TEXT | keep_original / transcode_keep / transcode_only |
| transcode_opts | TEXT (JSON) | codec/bitrate/application |
| min_free_bytes | INTEGER | free-space guard; refuse imports that would cross it |
| registered_at | TEXT (ISO8601) | |
| last_seen_at | TEXT (ISO8601) | updated each connect |

### 4.2 `recordings` — the backup ledger + pipeline source of truth
| Column | Type | Notes |
|--------|------|-------|
| id | TEXT PK | UUID |
| device_id | TEXT FK→devices | |
| source_path | TEXT | path on device at import time |
| library_path | TEXT | path in local library |
| sha256 | TEXT | of imported bytes; **dedupe key** |
| size_bytes | INTEGER | |
| duration_s | REAL | |
| codec | TEXT | original or `opus` |

| imported_at | TEXT (ISO8601) | |
| source_deleted | INTEGER (bool) | whether source was cleaned up |
| status | TEXT | imported / processing / done / quarantined / failed |
| UNIQUE | (device_id, sha256) | idempotent re-import guard |

### 4.6 `voiceprints` — cross-recording identity index
| Column | Type | Notes |
|--------|------|-------|
| id | INTEGER PK | |
| speaker_id | INTEGER FK→speakers | |

### 4.7 `diary_entries`
| Column | Type | Notes |
|--------|------|-------|

| date | TEXT (YYYY-MM-DD) | unique per local day |
| markdown_path | TEXT | |
| metadata | TEXT (JSON) | topics, mood, todos |

### 4.8 `diary_recordings` (join)
`(diary_entry_id, recording_id)` — which clips fed a day's entry.

### 4.9 `meetings`
| Column | Type | Notes |
|--------|------|-------|
| id | TEXT PK | |
| title | TEXT | LLM-generated |
| occurred_on | TEXT (YYYY-MM-DD) | for diary linking |
| markdown_path | TEXT | |
| metadata | TEXT (JSON) | decisions, action items |
| model / created_at | TEXT | |

### 4.10 `meeting_recordings` (join)
`(meeting_id, recording_id)` — groups contiguous clips into one meeting.

### 4.11 `meeting_participants` (join)
`(meeting_id, speaker_id, role?)`.

### 4.12 `diary_meeting_links`
`(diary_entry_id, meeting_id)` — the "see meeting" references inside a day.

### 4.13 `jobs` — durable pipeline queue
| Column | Type | Notes |
|--------|------|-------|
| id | INTEGER PK | |
| type | TEXT | transcode / stt / diarize / speaker_id / classify / diary_aggregate / meeting_extract / cloud_sync |
| payload | TEXT (JSON) | e.g. `{recording_id}` or `{date}` |
| status | TEXT | queued / running / done / failed / parked |
| attempts | INTEGER | retry count |
| worker_id | TEXT | claimer (crash reclaim) |
| last_error | TEXT | actionable message surfaced in UI |
| created_at / updated_at | TEXT | |

### 4.14 `cloud_sync_log`
`(id, remote, scope, started_at, finished_at, result, stats_json)`.

### 4.15 `quarantine`
`(id, device_id, source_path, quarantine_path, reason, size_bytes, detected_at)` —
a durable record of copies that failed SHA-256 verification (FR-2.7), so the event
is queryable and surfaceable in the UI rather than living only in the logs. The
source file is never deleted.

### 4.16 `schema_migrations`
`(version INTEGER, applied_at TEXT)` — drives automatic, versioned migrations
([FR-9.3](03-requirements-specification.md#fr-9-configuration--data-management)).

### 4.17 `ui_intents` (internal command queue)
An internal queue of user-triggered commands (`register_device`,
`rename_speaker`, `merge_speakers`, `sync_now`, `resummarize_day`). The **API
server** writes these when handling command endpoints
([11 §3](11-api-specification.md#3-rest-resources-v1)); the **daemon** consumes
them — keeping heavy work out of request handlers and the daemon decoupled from
the HTTP layer ([04 §3](04-system-architecture.md#3-processes)). Simple commands
may instead enqueue a `jobs` row directly; the distinction is an implementation
detail behind the API.

## 5. Indexing

- `recordings(device_id, sha256)` UNIQUE — dedupe.
- `segments(transcript_id, start_s)` — fast transcript rendering.
- `jobs(status, type)` — queue scans.
- FTS5 virtual table over `transcripts.text` (+ diary/meeting text) for
  [FR-7.5](03-requirements-specification.md#fr-7-user-interface) search.

## 6. Retention & integrity

- `staging/` holds only **pre-verification** copies; anything left there is debris
  from an interrupted run and is swept at the start of the next backup (the source
  on the device is still the authority, so nothing is lost).
- `quarantine/` copies are **never** auto-purged — they pair with a `quarantine`
  row (§4.15) and await user review; the source is never deleted.
- DB in WAL mode; periodic checkpoint; export/backup per
  [FR-9.4](03-requirements-specification.md#fr-9-configuration--data-management).
